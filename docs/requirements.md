# 要件定義

## 背景

利用者は、Androidスマホから自宅PC上のCodexへ開発指示を送りたい。実際の処理はPC側で進め、スマホはセッション選択、指示送信、状態確認、結果確認のためのリモート操作画面として使う。

## MVPゴール

MVPは、Xperia 1 IIIへAndroid APKをインストールし、次の流れを完了できる状態を目標とする。

1. Androidアプリを起動する。
2. 既存セッションを選択する、または新規セッションを作成する。
3. スマホから1件のテキスト指示を送信する。
4. 自宅PCが指示を受け取り、Codex処理を実行する。
5. アプリに処理状態、進捗概要、最終結果またはエラーが表示される。
6. 完了または失敗時にスマホへプッシュ通知が届く。

MVPでは、完全なターミナルストリーミング、スマホからPCへの直接接続、Google Play公開は対象外とする。現在のMVPでは、Codex実行中にPCブリッジが保存した定期的な進捗概要を表示してよい。

## 主な利用者

- Androidスマホと自宅PCの両方を所有する単一の開発者。
- 開発者は、PC上で実行される開発作業をスマホから遠隔操作したい。
- 開発者は、PCブリッジとFirebaseの初期セットアップを一度だけ実施できる前提とする。

## MVPプロダクト方針

- スマホとPCは、WiFiと携帯電話回線の両方で使えるように、インターネット経由のクラウドリレーで通信する。
- 自宅PCは起動している前提とする。
- MVPではVS CodeとCodex CLIがPC上で利用できる前提とする。
- PCブリッジからVS Codeを後から起動する機能は、別Issueで扱う拡張とする。
- Androidアプリは、最終結果と失敗理由に加えて、running中の進捗概要を表示してよい。完全な中間出力ストリーミングは対象外とする。
- MVPでは単一ユーザー、単一PCブリッジ、単一固定ワークスペースで十分とする。
- MVPの認証はFirebase identityを使った単一ユーザー向けの構成とする。
- MVPでは、PCブリッジのローカル設定に固定ワークスペースを持たせる。複数ワークスペース選択はpost-MVPとする。
- コマンド本文、進捗概要、最終結果は、認証とSecurity Rulesで保護したクラウドリレーに平文保存してよい。クライアント側暗号化はpost-MVPとする。
- Androidアプリは端末言語を自動認識し、日本語、英語、中国語、韓国語に対応する。未対応言語は英語にフォールバックする。

## 機能要件

### FR-001 セッション開始

Androidアプリは、新しいCodexリモートセッションを作成できなければならない。

受入条件:

- 利用者がアプリからセッションを作成できる。
- セッションがクラウドリレーに保存される。
- PCブリッジが新規セッションを検出できる。
- 利用者が理解できる既定タイトルが付く。

### FR-002 既存セッション選択

Androidアプリは、既存セッションを選択できなければならない。

受入条件:

- 認証済みユーザーに属するセッション一覧を表示できる。
- タイトル、更新時刻、状態など、セッションを識別できる情報を表示する。
- セッションを選択すると、コマンドと結果の画面を開ける。
- 他ユーザーのセッションは表示されない。

### FR-003 テキスト指示送信

Androidアプリは、選択中セッションへテキスト指示を送信できなければならない。

受入条件:

- 利用者がテキストを入力して送信できる。
- コマンドがPCブリッジ向けに `queued` として保存される。
- pending中の重複送信を防ぐ。
- 空のコマンドは送信できない。
- コマンド作成時刻を記録する。

### FR-004 進捗概要と最終結果表示

Androidアプリは、PC処理中の進捗概要と、完了後の最終結果を表示できなければならない。

受入条件:

- コマンドがrunning中の場合、PCブリッジが保存した定期的な進捗概要を表示できる。
- 最終応答テキストを表示できる。
- PCブリッジがエラーを報告した場合、失敗状態とエラー内容を表示できる。
- アプリ再起動後も最終結果を確認できる。

### FR-005 完了プッシュ通知

スマホは、処理完了または失敗時にプッシュ通知を受け取れなければならない。

受入条件:

- コマンド完了時にFCM通知が作成される。
- 通知をタップすると関連セッションを開ける。
- 失敗完了時も通知対象にできる。
- 通知payloadにはコマンド本文全文や結果全文を含めない。

### FR-006 PC起動前提

MVPではPCが起動している前提としてよい。

受入条件:

- PCブリッジがオフラインの場合、コマンドは `queued` のまま保持される。
- アプリはqueuedまたは待機状態を分かりやすく表示する。
- PCブリッジが `running` にするまで、アプリは処理開始済みと誤表示しない。

### FR-007 VS Code/Codex利用前提と任意起動

MVPではVS CodeとCodex CLIが利用できる前提としてよい。PCブリッジからVS Codeを起動する機能は後続拡張とする。

受入条件:

- MVPでは、PC側でCodex CLIを実行できる状態でコマンド処理できる。
- VS Code自動起動は別Issueとして追跡する。
- VS CodeまたはCodexが利用できない場合、コマンドは見えるエラー状態になる。

### FR-008 Xperia 1 III対応

AndroidアプリはXperia 1 IIIで利用できなければならない。

受入条件:

- Androidのmin SDKとtarget SDKがXperia 1 IIIをサポートする。
- 21:9の縦画面で主要レイアウトが破綻しない。
- APKを実機へインストールして起動できる。
- テキスト入力、セッション一覧、結果表示が縦画面で使える。

### FR-009 WiFiと携帯電話回線

アプリはWiFiと携帯電話回線の両方で使えなければならない。

受入条件:

- 中核通信はインターネット到達可能なクラウドリレーを使う。
- 自宅ルーターのポート開放を不要にする。
- LAN直結モードは後から追加してよいが、MVP必須ではない。

### FR-010 デバイス認証とペアリング

未知のスマホや未知のPCブリッジがセッションへアクセスできないようにする。

受入条件:

- Androidアプリは認証済みユーザーまたは端末IDを持つ。
- PCブリッジは同じユーザーに紐づく登録済みIDを持つ。
- MVPでは単一ユーザー向けの簡易ペアリングまたは手動設定を許容する。
- ペアリング情報やcredentialはGitに保存しない。

### FR-011 ワークスペース対象の明示

利用者は、どのPCまたはワークスペースへ指示が送られるか理解できなければならない。

受入条件:

- 各コマンドに対象PCブリッジを記録する。
- アプリはPCブリッジ名など、読める対象名を表示する。
- MVPで複数ワークスペース選択を持たない場合、PCブリッジ設定に固定ワークスペースを明記する。

### FR-012 コマンドライフサイクル

コマンドは明示的な状態を持たなければならない。

受入条件:

- 状態は `queued`, `running`, `completed`, `failed`, `canceled` をサポートする。
- Androidアプリは `queued`, `running`, `completed`, `failed` を表示する。
- `canceled` は、UI未実装でもデータモデル上予約してよい。
- PCブリッジは開始時刻と完了時刻を可能な範囲で記録する。

### FR-013 セットアップ状態の可視化

アプリは、よくあるセットアップ不備を利用者が理解できるように表示しなければならない。

受入条件:

- PCブリッジ未登録の場合、セットアップが必要な状態を示す。
- PCブリッジ登録済みだが非アクティブの場合、待機中またはオフラインを示す。
- 通知権限がない場合、権限要求または説明を表示する。

### FR-014 端末言語による表示切替

Androidアプリは、スマホの言語設定と表示設定に合わせて言語とテーマを切り替えられなければならない。

受入条件:

- 日本語、英語、中国語、韓国語に対応する。
- 未対応言語は英語へフォールバックする。
- Androidのシステムテーマがダークの場合、アプリもダークテーマで表示する。
- Androidのシステムテーマがライトの場合、アプリもライトテーマで表示する。
- セッション、コマンド、PCブリッジ状態、設定、CLIオプションヘルプなど主要UIを翻訳する。
- CLIオプション名、model名、API名、ファイルパスなどの技術識別子は翻訳しない。
- 少なくとも1つの非英語localeをwidget testで確認する。

### FR-015 セッション管理

Androidアプリは、過去セッションをスマホ上で整理して再利用できなければならない。

受入条件:

- セッション名をアプリ上で変更できる。
- セッション名で一覧を検索できる。
- セッションをお気に入り登録し、お気に入りが一覧で見分けられる。
- セッションに任意のグループ名を付け、グループで絞り込める。
- セッション履歴をアプリから削除できる。
- 削除は初期実装では `deletedAt` を使ったソフト削除とし、PCブリッジ処理履歴の物理削除は行わない。

### FR-016 post-MVP 添付とPC電源投入

セッション途中の添付設計は [Session attachment flow design](attachment-flow-design.md) に分離する。Wake-on-LAN/PC電源投入は別Issueとして扱う。

セッション開始後のファイル添付と、PC電源オフ状態からの起動はpost-MVP拡張として扱う。

受入条件:

- 途中添付は、スマホ上のファイルをPCブリッジから参照可能にする保存先、権限、転送方式を設計してから実装する。
- 画像添付はCodex CLIの `--image` とPCブリッジの `codexImages` 入力を前提に検討する。
- 汎用ファイル添付は、PC側ワークスペースへの配置、クラウドストレージ、またはテキスト指示内の参照パスのどれを採用するか別Issueで決める。
- PC電源投入はWake-on-LAN、BIOS/UEFI設定、ネットワーク越え、常時稼働中継端末の有無を確認してから実装する。

## 非機能要件

### NFR-001 セキュリティ

- 認証済み端末だけがセッションへアクセスできる。
- コマンドは認証済みユーザーにスコープされる。
- PCブリッジは認証なしの任意実行口を公開しない。
- 秘密情報はリポジトリへコミットしない。
- PCブリッジはスマホ入力をraw shell commandではなくCodexへの指示として扱う。
- Firestore Security Rulesまたは同等のリレー権限をテストする。

### NFR-002 信頼性

- 一時的にスマホが切断されてもコマンドが失われない。
- PCブリッジは一時的なリレー障害を再試行できる。
- コマンド状態は `queued`, `running`, `completed`, `failed`, `canceled` として明示する。
- 複数ブリッジや再試行により、同じコマンドを二重処理しない。
- アプリ再起動後もセッション状態を復元できる。

### NFR-003 プライバシー

- セッション本文と最終結果には、ソースコードや非公開情報が含まれる可能性がある。
- 公開リリース前にデータ保持方針を文書化する。
- 通知payloadには秘匿性の高い本文全文を含めない。
- ログには秘密情報を保存しない。

### NFR-004 保守性

- Androidアプリ、クラウドデータモデル、PCブリッジの境界を明確にする。
- コマンドライフサイクルはテストで確認する。
- 工程別作業はGitHub Issueで追跡する。

### NFR-005 使いやすさ

- アプリは縦画面で片手操作しやすい構成にする。
- 長い結果はスクロールできる。
- 送信ボタンや状態表示から、pending/runningが分かる。

### NFR-006 観測性

- PCブリッジはセットアップや実行時診断用のローカルログを出す。
- コマンド失敗時のエラーは、秘密情報を含めず次の確認行動が分かる内容にする。
- Release検証では、アプリversion、build number、commit SHA、対象端末を記録する。

## MVP受入シナリオ

前提:

- AndroidアプリがXperia 1 IIIにインストールされている。
- スマホがWiFiまたは携帯電話回線でインターネットに接続できる。
- PCが起動している。
- PCブリッジが自宅PC上で起動している。
- スマホとPCブリッジが同じユーザーに紐づいている。

操作:

1. 利用者がAndroidアプリを開く。
2. 利用者がセッションを作成または選択する。
3. 利用者がテキスト指示を送信する。
4. PCブリッジが指示を受け取り、Codexへ渡す。

期待結果:

- コマンド状態が `queued -> running -> completed`、または失敗時は `failed` へ進む。
- アプリに進捗概要、最終結果、またはエラーが表示される。
- 完了または失敗時にスマホへプッシュ通知が届く。
- 自宅ルーターのポート開放なしで動作する。

## MVP対象外

- 完全な中間ターミナル出力ストリーミング。
- スマホからの直接ファイル編集。
- 携帯電話回線からのWake-on-LAN。
- 複数ユーザーのチーム利用。
- Marketplace公開。
- Google Play Store公開。
- Codexセッション transcript の詳細閲覧。
- 複数PCブリッジの本格ルーティング。
- リモートデスクトップやVS Code UIのミラーリング。
- ARBや翻訳サービスを使った本格的な翻訳管理。MVPでは軽量なアプリ内文字列テーブルでよい。

## アーキテクチャ工程への引き継ぎ事項

- ペアリングをFirebase anonymous auth、email sign-in、その他Firebase対応認証のどれで扱うか。
- PCブリッジ登録を短時間pairing code、ローカル設定secret、またはその両方で扱うか。
- コマンドと結果アクセスのFirestore Security Rulesとテスト。
- コマンドと結果documentの保持期間。

## QCDS強化要件

Release 1.0.2以降の品質、コスト、デリバリー、セキュリティ改善は [QCDS強化計画](qcds-hardening-plan.md) と [仕様書](specification.md) に定義する。

主な追加要件:

- Quality: Flutter widget/golden test、Android主要導線回帰テスト、PCブリッジclaim/timeout/redactionテスト、Firebase Emulatorテスト、実機E2E手順化、診断ログ収集手順を整備する。
- Cost: 自前Firebase方式を維持し、セットアップWeb UIの設定チェック、`config.local.json` 生成補助、PCブリッジ運用確認、Firebase費用注意点を追加する。
- Delivery: Release Runbook、自動artifact生成補助、CI build/test/audit、Release前チェックを整備する。
- Security: Publicリポジトリ向け監査、秘密情報漏えい時Runbook、Firebase service account最小権限方針を整備する。
